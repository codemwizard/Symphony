# Phase-3 Agentic SDLC Policy

## Phase Scope

This policy defines the agentic software development lifecycle requirements and
operating rules specifically for **Phase-3: Constraint and Legitimacy Engine**
of the Symphony system.

**Phase-3 Status**: Activation in progress. The lifecycle artifact set exists to
support opening and verification work, while the root execution envelope remains
the controlling authority for what is currently executable.

## Authority Hierarchy

### Apex Authority
- **Operation Manual**: `docs/operations/AI_AGENT_OPERATION_MANUAL.md`
  - Canonical authority for agent operation rules and role definitions

### Phase Taxonomy Authority
- **Lifecycle Policy**: `docs/operations/PHASE_LIFECYCLE.md`
  - Authority for phase boundaries, required artifacts, and transition rules

### Phase-3 Specific Authority
- **Machine Contract**: `docs/PHASE3/phase3_contract.yml`
  - Authoritative for Phase-3 delivery claims
- **Verifier**: `scripts/audit/verify_phase3_contract.sh`
  - Canonical lifecycle artifact verifier for this activation stage
- **Evidence Namespace**: `evidence/phase3/**`
  - Required namespace for Phase-3 evidence artifacts

## Phase-3 Claim Requirements

### Required Evidence for Phase-3 Delivery Claims
All Phase-3 delivery claims MUST be supported by:

1. **Machine Contract Rows**
   - Must remain aligned to `docs/PHASE3/phase3_contract.yml`
   - Must use Phase-3 scope and terminology only
   - Must not claim opened execution surface by implication

2. **Verifier Output**
   - `scripts/audit/verify_phase3_contract.sh` must emit PASS status
   - Evidence must be generated at
     `evidence/phase3/tsk_p3_act_001_lifecycle_artifacts.json`
   - Verifier must execute with fail-closed behavior

3. **Phase-3 Evidence**
   - All Phase-3 evidence must live under `evidence/phase3/**`
   - Evidence must include `task_id`, `git_sha`, `timestamp_utc`, and `status`
   - Evidence must be deterministic and reproducible

### Claim Discipline Rules
- **Machine-Contract First**: Delivery claims must defer to
  `docs/PHASE3/phase3_contract.yml`
- **Evidence Required**: Every delivery claim must have verifier-backed evidence
- **Envelope Discipline**: No claim may state that Phase-3 execution is active
  unless the root execution envelope has been updated accordingly
- **Fail-Closed Verification**: Claims must survive fail-closed verification

## Phase-3 Execution Rules

### Agent Role Boundaries
- **ARCHITECT**: May create human contracts, planning, and activation doctrine-aligned documentation
- **SECURITY_GUARDIAN**: May create regulated verifiers and policy guards under `scripts/audit/**` and `docs/operations/**`
- **INVARIANTS_CURATOR**: May maintain invariant documentation where Phase-3 doctrine requires it
- **DB_FOUNDATION**: May implement database-layer Phase-3 work only after activation advances to executable task scope

### Evidence Requirements
- All Phase-3 work must generate evidence in `evidence/phase3/**`
- Evidence must be JSON-formatted with required schema fields
- Evidence must be version-controlled and reproducible

### Verification Requirements
- Phase-3 lifecycle artifact changes must pass `verify_phase3_contract.sh`
- Local development should use `RUN_PHASE3_GATES=1 bash scripts/dev/pre_ci.sh`
  when the activation sequence requires full parity checks
- CI wiring for Phase-3 gates follows the activation sequence; do not overclaim
  CI integration before the envelope and legality layers are reconciled

## Anti-Drift Measures

### Contract Integrity
- `docs/PHASE3/phase3_contract.yml` is the authoritative source for
  delivery-claimable Phase-3 requirements
- `docs/PHASE3/PHASE3_CONTRACT.md` is explanatory and must not introduce new
  delivery claims
- Any conflict must be resolved in favor of the machine contract and the root
  execution envelope

### Evidence Integrity
- Evidence must be generated by canonical verifiers
- Evidence must reference specific git commits for reproducibility
- Pre-opening historical Phase-3 evidence must not be silently treated as
  opened-phase delivery proof

### Process Integrity
- Activation tasks must follow the declared activation order
- No Phase-3 task may claim broader execution authority than the current
  envelope permits
- Regulated-surface edits require Stage A approval before implementation

## Phase-3 Completion Criteria

Phase-3 is considered complete only when:
1. Required machine-contract rows pass under `RUN_PHASE3_GATES=1`
2. Verifier and evidence coverage are complete
3. Envelope, legality, and planning posture are reconciled
4. Deferred capability rows are explicitly staged forward as needed

## Phase-3 Limitations

### Explicitly Out of Scope
- Future-phase capability implementation
- Envelope rewrite by implication rather than governed change
- Delivery claims without verifier-backed evidence
- Silent reclassification of historical planning evidence

### Forbidden Actions
- Bypassing approval workflow for regulated surfaces
- Treating the human contract as authoritative over the machine contract
- Claiming active execution surface status before the envelope says so
- Using Phase-3 evidence outside `evidence/phase3/**`

## Compliance and Audit

### Audit Requirements
- All Phase-3 activation work must be auditable through approval and evidence trails
- Audit trails must include task IDs, git commits, timestamps, and approval references

### Compliance Checks
- Regular verification of contract alignment via `verify_phase3_contract.sh`
- Validation of approval metadata before regulated edits
- Confirmation that `RUN_PHASE3_GATES=1` and `evidence/phase3/**` remain aligned across policy and contract artifacts
