# Phase-2 Agentic SDLC Policy

## Phase Scope

This policy defines the agentic software development lifecycle (SDLC) requirements and operating rules specifically for **Phase-2: Internal Ledger Truth** of the Symphony system.

**Phase-2 Status**: **CLOSED** - Phase-2 is not open for new development work except through explicit governance convergence tasks.

## Authority Hierarchy

### Apex Authority
- **Operation Manual**: `docs/operations/AI_AGENT_OPERATION_MANUAL.md`
  - Contains the canonical agent operation rules and role definitions
  - Apex authority for all agentic SDLC activities

### Phase Taxonomy Authority  
- **Lifecycle Policy**: `docs/operations/PHASE_LIFECYCLE.md`
  - Defines phase boundaries, completion criteria, and transition rules
  - Authority for phase-level governance decisions

### Phase-2 Specific Authority
- **Machine Contract**: `docs/PHASE2/phase2_contract.yml`
  - Invariant-centric contract rows define delivery requirements
  - Authoritative for Phase-2 delivery claims

## Phase-2 Claim Requirements

### Required Evidence for Phase-2 Delivery Claims
All Phase-2 delivery claims MUST be supported by:

1. **Machine Contract Rows**
   - Must reference registered invariant IDs (INV-XXX format)
   - Must have `status: implemented` for delivery claims
   - Must include verifier and evidence path references

2. **Verifier Output**
   - `scripts/audit/verify_phase2_contract.sh` must emit PASS status
   - Evidence must be generated at `evidence/phase2/phase2_contract_status.json`
   - Verifier must be executed with fail-closed behavior

3. **Phase-2 Evidence**
   - All evidence must be stored under `evidence/phase2/**`
   - Evidence files must include `task_id`, `git_sha`, `timestamp_utc`, and `status`
   - Evidence must be deterministic and reproducible

### Claim Discipline Rules
- **No Task-ID Claims**: Delivery claims cannot be based on task IDs (TSK-P2-XXX format)
- **Invariant-Centric Only**: Only registered invariant IDs may support delivery claims
- **Evidence Required**: Every delivery claim must have corresponding verifier evidence
- **Fail-Closed Verification**: Claims must survive fail-closed verification

## Phase-2 Execution Rules

### Agent Role Boundaries
- **INVARIANTS_CURATOR**: May register invariants and update machine contract
- **SECURITY_GUARDIAN**: May create verifiers and wire CI gates
- **ARCHITECT**: May create human contracts and policy documents
- **DB_FOUNDATION**: May implement database changes for registered invariants

### Evidence Requirements
- All Phase-2 work must generate evidence in `evidence/phase2/**`
- Evidence must be JSON-formatted with required schema fields
- Evidence must be version-controlled and reproducible

### Verification Requirements
- All Phase-2 changes must pass `verify_phase2_contract.sh`
- Local development must use `RUN_PHASE2_GATES=1 scripts/dev/pre_ci.sh`
- CI must run Phase-2 contract verification with fail-closed behavior

## Anti-Drift Measures

### Contract Integrity
- Machine contract (`phase2_contract.yml`) is the sole authority for delivery claims
- Human documents are explanatory only and must not introduce new requirements
- Any conflicts between documents must be resolved in favor of the machine contract

### Evidence Integrity
- Evidence files must be signed and versioned
- Evidence must be generated by canonical verifiers
- Evidence must reference specific git commits for reproducibility

### Process Integrity
- All Phase-2 work must follow the sequential task order
- No task may be skipped or executed out of sequence
- All prerequisite tasks must be completed before proceeding

## Phase-2 Completion Criteria

Phase-2 is considered complete when:
1. All registered invariants have `status: implemented`
2. All verifiers pass with fail-closed behavior
3. All evidence is generated and validated
4. Human/machine contract alignment is verified
5. CI gates are wired and functional

## Phase-2 Limitations

### Explicitly Out of Scope
- **New Feature Development**: Phase-2 is closed to new features
- **External Integrations**: No new external system integrations
- **Phase-3 Work**: Phase-3 work is explicitly deferred
- **Policy Changes**: No changes to higher-level policy documents

### Forbidden Actions
- **Direct Machine Contract Edits**: Must go through governance convergence tasks
- **Bypassing Verification**: All changes must survive verification
- **Missing Evidence**: No delivery claims without supporting evidence
- **Role Violations**: Agents must stay within defined role boundaries

## Phase-2 Governance

### Review Requirements
- All Phase-2 changes require Stage A approval before implementation
- All Phase-2 changes require Stage B approval after PR creation
- Changes to regulated surfaces require explicit approval metadata

### Escalation Paths
- **Verification Failures**: Must be resolved through remediation casefiles
- **Policy Conflicts**: Must be escalated to apex authority documents
- **Role Boundary Issues**: Must be resolved through agent governance process

## Compliance and Audit

### Audit Requirements
- All Phase-2 work must be auditable through evidence trails
- Evidence must be retained for compliance verification
- Audit trails must include task IDs, git commits, and timestamps

### Compliance Checks
- Regular verification of contract alignment
- Periodic audit of evidence completeness
- Validation of role boundary compliance
