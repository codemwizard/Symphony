# Symphony Adjustment Layer State-Transition Package (Draft v1)

## 0) Scope and assumptions

This package governs post-terminal actions through these entities:

* `adjustment_instruction`
* `adjustment_approval`
* `adjustment_execution`

It assumes:

* base `instruction` table exists (original transaction/instruction lifecycle)
* terminal immutability on `instruction` enforced via `P7101`
* recipient-change exception workflow exists (`recipient_change_exception`)
* evidence emission exists (or can be invoked via app/service layer)
* RLS / tenant isolation exists

# 2) Legal state transitions (adjustment_instruction)

## 2.1 Transition matrix (canonical)

| From                       | To                         | Allowed?    | Condition                                          |
| -------------------------- | -------------------------- | ----------- | -------------------------------------------------- |
| `DRAFT`                    | `PENDING_APPROVAL`         | Yes         | required fields valid                              |
| `DRAFT`                    | `CANCELLED`                | Yes         | requester/admin policy                             |
| `DRAFT`                    | `REJECTED`                 | No          | must go through approval path                      |
| `PENDING_APPROVAL`         | `APPROVED_PENDING_COOLING` | Yes         | approvals quorum satisfied + cooling required      |
| `PENDING_APPROVAL`         | `ELIGIBLE_TO_EXECUTE`      | Yes         | approvals quorum satisfied + no cooling required   |
| `PENDING_APPROVAL`         | `REJECTED`                 | Yes         | any required approver rejects (policy-dependent)   |
| `PENDING_APPROVAL`         | `CANCELLED`                | Yes         | before final approval, authorized role             |
| `APPROVED_PENDING_COOLING` | `ELIGIBLE_TO_EXECUTE`      | Yes         | cooling expired + approvals still valid            |
| `APPROVED_PENDING_COOLING` | `CANCELLED`                | Yes         | authorized role before execution                   |
| `APPROVED_PENDING_COOLING` | `EXPIRED`                  | Yes         | policy TTL elapsed                                 |
| `ELIGIBLE_TO_EXECUTE`      | `EXECUTION_IN_PROGRESS`    | Yes         | execution row inserted / dispatch begins           |
| `ELIGIBLE_TO_EXECUTE`      | `CANCELLED`                | Yes         | authorized role before execution begins            |
| `ELIGIBLE_TO_EXECUTE`      | `EXPIRED`                  | Yes         | policy TTL elapsed                                 |
| `EXECUTION_IN_PROGRESS`    | `EXECUTED`                 | Yes         | execution outcome terminal success                 |
| `EXECUTION_IN_PROGRESS`    | `REJECTED`                 | No          | execution has started; use execution outcome       |
| `EXECUTION_IN_PROGRESS`    | `CANCELLED`                | No          | use execution containment/failure                  |
| `EXECUTION_IN_PROGRESS`    | `ELIGIBLE_TO_EXECUTE`      | Optional/No | avoid oscillation; prefer execution attempts table |
| `EXECUTED`                 | *any*                      | No          | terminal immutable                                 |
| `REJECTED`                 | *any*                      | No          | terminal immutable                                 |
| `CANCELLED`                | *any*                      | No          | terminal immutable                                 |
| `EXPIRED`                  | *any*                      | No          | terminal immutable                                 |

# 3) Core invariants (hard constraints)

## 3.1 Parent transaction immutability

* Original `instruction` terminal rows must never be updated for financial/finality changes.
* All post-terminal handling uses linked adjustment rows.

## 3.2 Recipient continuity default

* `proposed_recipient_ref_hash = original_recipient_ref_hash` unless approved `recipient_change_exception`.

## 3.3 Maker-checker separation

* Requester cannot satisfy all approvals.
* Same actor cannot approve multiple stages if policy forbids.

## 3.4 Cooling-off enforcement

* If `cooling_required = true`, execution cannot begin before `cooling_until`.

## 3.5 Approval quorum snapshot integrity

* Approval requirements evaluated against the policy snapshot stored on the adjustment (`policy_id`, `policy_version`) — not “current policy at runtime”.

## 3.6 Terminal immutability (adjustment instruction)

* Terminal `adjustment_status` rows cannot be mutated (except explicit allowlisted non-business metadata if absolutely required).

## 6.4 `fn_adjustment_approval_quorum_satisfied(adjustment_id uuid)`

Checks that all required stages have approved decisions.

```sql
CREATE OR REPLACE FUNCTION fn_adjustment_approval_quorum_satisfied(p_adjustment_id UUID)
RETURNS BOOLEAN
LANGUAGE plpgsql
STABLE
AS $$
DECLARE
    v_missing_count INTEGER;
BEGIN
    -- If any rejection exists, quorum is considered not satisfied
    IF fn_adjustment_has_rejection(p_adjustment_id) THEN
        RETURN FALSE;
    END IF;

    WITH required AS (
        SELECT stage FROM fn_required_approval_stages(p_adjustment_id)
    ),
    approved AS (
        SELECT DISTINCT approval_stage AS stage
        FROM adjustment_approval
        WHERE adjustment_id = p_adjustment_id
          AND approval_decision = 'APPROVED'
    )
    SELECT COUNT(*)
      INTO v_missing_count
    FROM required r
    LEFT JOIN approved a USING (stage)
    WHERE a.stage IS NULL;

    RETURN v_missing_count = 0;
END;
$$;
```

## 6.5 `fn_adjustment_maker_checker_valid(adjustment_id uuid)`

Enforces requester ≠ approvers and optionally no same actor across multiple stages.

```sql
CREATE OR REPLACE FUNCTION fn_adjustment_maker_checker_valid(p_adjustment_id UUID)
RETURNS BOOLEAN
LANGUAGE plpgsql
STABLE
AS $$
DECLARE
    v_requester UUID;
    v_violation BOOLEAN;
BEGIN
    SELECT request_actor_id INTO v_requester
    FROM adjustment_instruction
    WHERE adjustment_id = p_adjustment_id;

    IF NOT FOUND THEN
        RETURN FALSE;
    END IF;

    -- requester cannot approve any stage
    SELECT EXISTS (
        SELECT 1
        FROM adjustment_approval aa
        WHERE aa.adjustment_id = p_adjustment_id
          AND aa.approver_actor_id = v_requester
    ) INTO v_violation;

    IF v_violation THEN
        RETURN FALSE;
    END IF;

    -- Optional stricter rule: same actor cannot approve multiple stages
    SELECT EXISTS (
        SELECT 1
        FROM (
            SELECT approver_actor_id
            FROM adjustment_approval
            WHERE adjustment_id = p_adjustment_id
              AND approval_decision = 'APPROVED'
            GROUP BY approver_actor_id
            HAVING COUNT(DISTINCT approval_stage) > 1
        ) x
    ) INTO v_violation;

    IF v_violation THEN
        RETURN FALSE;
    END IF;

    RETURN TRUE;
END;
$$;
```

## 7.1 Parent terminal immutability (`instruction`) — `P7101`

(From prior draft, included here as part of package.)

```sql
CREATE OR REPLACE FUNCTION trg_instruction_terminal_immutability()
RETURNS trigger
LANGUAGE plpgsql
AS $$
DECLARE
    v_old_terminal BOOLEAN;
BEGIN
    v_old_terminal := OLD.current_state IN ('FINAL_CONFIRMED','CLOSED_RESOLVED','CLOSED_UNRESOLVED');

    IF v_old_terminal THEN
        -- Protect core business fields. Expand this list explicitly.
        IF (NEW.current_state IS DISTINCT FROM OLD.current_state)
           OR (NEW.amount_minor IS DISTINCT FROM OLD.amount_minor)
           OR (NEW.currency_code IS DISTINCT FROM OLD.currency_code)
           OR (NEW.recipient_ref_hash IS DISTINCT FROM OLD.recipient_ref_hash)
           OR (NEW.finality_reference IS DISTINCT FROM OLD.finality_reference)
        THEN
            RAISE EXCEPTION USING
                ERRCODE = 'P7101',
                MESSAGE = 'TERMINAL_RECORD_IMMUTABLE_USE_ADJUSTMENT_LAYER';
        END IF;
    END IF;

    RETURN NEW;
END;
$$;

CREATE TRIGGER instruction_terminal_immutability
BEFORE UPDATE ON instruction
FOR EACH ROW
EXECUTE FUNCTION trg_instruction_terminal_immutability();
```

## 7.2 `adjustment_instruction` status + immutability + basic guards

This trigger validates legal state transitions and terminal immutability of the adjustment itself.

```sql
CREATE OR REPLACE FUNCTION trg_adjustment_instruction_state_guard()
RETURNS trigger
LANGUAGE plpgsql
AS $$
DECLARE
    v_old_terminal BOOLEAN;
    v_parent_state TEXT;
BEGIN
    IF TG_OP = 'INSERT' THEN
        -- Initial state must be DRAFT
        IF NEW.adjustment_status <> 'DRAFT' THEN
            RAISE EXCEPTION USING
                ERRCODE = 'P7401',
                MESSAGE = 'INITIAL_ADJUSTMENT_STATE_MUST_BE_DRAFT';
        END IF;

        -- Optional: require parent instruction to be terminal before adjustment
        SELECT current_state INTO v_parent_state
        FROM instruction
        WHERE instruction_id = NEW.parent_instruction_id
          AND tenant_id = NEW.tenant_id;

        IF NOT FOUND THEN
            RAISE EXCEPTION USING
                ERRCODE = 'P7403',
                MESSAGE = 'ADJUSTMENT_PARENT_NOT_FOUND';
        END IF;

        IF v_parent_state NOT IN ('FINAL_CONFIRMED','CLOSED_RESOLVED','CLOSED_UNRESOLVED') THEN
            RAISE EXCEPTION USING
                ERRCODE = 'P7403',
                MESSAGE = 'ADJUSTMENT_PARENT_NOT_TERMINAL';
        END IF;

        NEW.updated_at := now();
        RETURN NEW;
    END IF;

    -- UPDATE path
    v_old_terminal := fn_is_adjustment_terminal(OLD.adjustment_status);

    IF v_old_terminal THEN
        -- Terminal adjustment records immutable (except optional metadata allowlist)
        IF (NEW.adjustment_status IS DISTINCT FROM OLD.adjustment_status)
           OR (NEW.requested_amount_minor IS DISTINCT FROM OLD.requested_amount_minor)
           OR (NEW.proposed_recipient_ref_hash IS DISTINCT FROM OLD.proposed_recipient_ref_hash)
           OR (NEW.reason_code IS DISTINCT FROM OLD.reason_code)
        THEN
            RAISE EXCEPTION USING
                ERRCODE = 'P7402',
                MESSAGE = 'ADJUSTMENT_TERMINAL_IMMUTABLE';
        END IF;

        NEW.updated_at := OLD.updated_at;
        RETURN NEW;
    END IF;

    -- Legal transition enforcement
    IF NEW.adjustment_status IS DISTINCT FROM OLD.adjustment_status THEN
        PERFORM 1 FROM (
            VALUES
              ('DRAFT','PENDING_APPROVAL'),
              ('DRAFT','CANCELLED'),
              ('PENDING_APPROVAL','APPROVED_PENDING_COOLING'),
              ('PENDING_APPROVAL','ELIGIBLE_TO_EXECUTE'),
              ('PENDING_APPROVAL','REJECTED'),
              ('PENDING_APPROVAL','CANCELLED'),
              ('APPROVED_PENDING_COOLING','ELIGIBLE_TO_EXECUTE'),
              ('APPROVED_PENDING_COOLING','CANCELLED'),
              ('APPROVED_PENDING_COOLING','EXPIRED'),
              ('ELIGIBLE_TO_EXECUTE','EXECUTION_IN_PROGRESS'),
              ('ELIGIBLE_TO_EXECUTE','CANCELLED'),
              ('ELIGIBLE_TO_EXECUTE','EXPIRED'),
              ('EXECUTION_IN_PROGRESS','EXECUTED')
        ) AS t(from_state, to_state)
        WHERE t.from_state = OLD.adjustment_status
          AND t.to_state = NEW.adjustment_status;

        IF NOT FOUND THEN
            RAISE EXCEPTION USING
                ERRCODE = 'P7401',
                MESSAGE = format('INVALID_ADJUSTMENT_STATE_TRANSITION: %s -> %s', OLD.adjustment_status, NEW.adjustment_status);
        END IF;
    END IF;

    NEW.updated_at := now();
    RETURN NEW;
END;
$$;

CREATE TRIGGER adjustment_instruction_state_guard
BEFORE INSERT OR UPDATE ON adjustment_instruction
FOR EACH ROW
EXECUTE FUNCTION trg_adjustment_instruction_state_guard();
```

## 7.3 Recipient binding enforcement (no silent destination drift)

(Refined from prior draft.)

```sql
CREATE OR REPLACE FUNCTION trg_adjustment_enforce_recipient_binding()
RETURNS trigger
LANGUAGE plpgsql
AS $$
DECLARE
    v_exception recipient_change_exception%ROWTYPE;
BEGIN
    -- only enforce on INSERT or if recipient-related fields attempted to change
    IF TG_OP = 'UPDATE' THEN
        IF NEW.proposed_recipient_ref_hash IS NOT DISTINCT FROM OLD.proposed_recipient_ref_hash
           AND NEW.recipient_change_exception_id IS NOT DISTINCT FROM OLD.recipient_change_exception_id
        THEN
            RETURN NEW;
        END IF;
    END IF;

    IF NEW.proposed_recipient_ref_hash <> NEW.original_recipient_ref_hash THEN
        IF NEW.recipient_change_exception_id IS NULL THEN
            RAISE EXCEPTION USING ERRCODE = 'P7202',
                MESSAGE = 'RECIPIENT_CHANGE_NOT_ALLOWED_WITHOUT_APPROVED_EXCEPTION';
        END IF;

        SELECT * INTO v_exception
        FROM recipient_change_exception
        WHERE recipient_change_exception_id = NEW.recipient_change_exception_id
          AND tenant_id = NEW.tenant_id
          AND parent_instruction_id = NEW.parent_instruction_id
          AND status = 'APPROVED';

        IF NOT FOUND THEN
            RAISE EXCEPTION USING ERRCODE = 'P7203',
                MESSAGE = 'RECIPIENT_CHANGE_EXCEPTION_NOT_APPROVED_OR_INVALID';
        END IF;

        IF v_exception.expires_at IS NOT NULL AND now() > v_exception.expires_at THEN
            RAISE EXCEPTION USING ERRCODE = 'P7204',
                MESSAGE = 'RECIPIENT_CHANGE_EXCEPTION_EXPIRED';
        END IF;
    END IF;

    RETURN NEW;
END;
$$;

CREATE TRIGGER adjustment_enforce_recipient_binding
BEFORE INSERT OR UPDATE ON adjustment_instruction
FOR EACH ROW
EXECUTE FUNCTION trg_adjustment_enforce_recipient_binding();
```

## 7.5 Approval insert guard (state + maker-checker + stage validity)

This prevents unauthorized approval writes and bad stage sequencing.

```sql
CREATE OR REPLACE FUNCTION trg_adjustment_approval_guard()
RETURNS trigger
LANGUAGE plpgsql
AS $$
DECLARE
    v_adj adjustment_instruction%ROWTYPE;
    v_requester UUID;
    v_stage_required BOOLEAN;
    v_existing_stage_approval BOOLEAN;
BEGIN
    SELECT * INTO v_adj
    FROM adjustment_instruction
    WHERE adjustment_id = NEW.adjustment_id
    FOR UPDATE;

    IF NOT FOUND THEN
        RAISE EXCEPTION USING ERRCODE = 'P7410', MESSAGE = 'ADJUSTMENT_NOT_FOUND';
    END IF;

    IF v_adj.adjustment_status NOT IN ('PENDING_APPROVAL','APPROVED_PENDING_COOLING') THEN
        RAISE EXCEPTION USING ERRCODE = 'P7410',
            MESSAGE = 'ADJUSTMENT_APPROVAL_NOT_ALLOWED_IN_STATE';
    END IF;

    -- Reject adding approvals after terminal adjustment state
    IF fn_is_adjustment_terminal(v_adj.adjustment_status) THEN
        RAISE EXCEPTION USING ERRCODE = 'P7410',
            MESSAGE = 'ADJUSTMENT_APPROVAL_NOT_ALLOWED_FOR_TERMINAL_RECORD';
    END IF;

    -- Maker-checker: requester cannot approve
    IF NEW.approver_actor_id = v_adj.request_actor_id THEN
        RAISE EXCEPTION USING ERRCODE = 'P7411',
            MESSAGE = 'MAKER_CHECKER_VIOLATION';
    END IF;

    -- Stage must be required by policy snapshot
    SELECT EXISTS (
        SELECT 1
        FROM fn_required_approval_stages(NEW.adjustment_id) s
        WHERE s.stage = NEW.approval_stage
    ) INTO v_stage_required;

    IF NOT v_stage_required THEN
        RAISE EXCEPTION USING ERRCODE = 'P7414',
            MESSAGE = 'APPROVAL_STAGE_NOT_REQUIRED';
    END IF;

    -- Prevent duplicate APPROVED stage decisions (policy can be tuned)
    SELECT EXISTS (
        SELECT 1
        FROM adjustment_approval aa
        WHERE aa.adjustment_id = NEW.adjustment_id
          AND aa.approval_stage = NEW.approval_stage
          AND aa.approval_decision = 'APPROVED'
    ) INTO v_existing_stage_approval;

    IF v_existing_stage_approval AND NEW.approval_decision = 'APPROVED' THEN
        RAISE EXCEPTION USING ERRCODE = 'P7412',
            MESSAGE = 'DUPLICATE_APPROVAL_STAGE';
    END IF;

    -- Optional strict same-actor multi-stage restriction (for APPROVED decisions)
    IF NEW.approval_decision = 'APPROVED' THEN
        IF EXISTS (
            SELECT 1
            FROM adjustment_approval aa
            WHERE aa.adjustment_id = NEW.adjustment_id
              AND aa.approver_actor_id = NEW.approver_actor_id
              AND aa.approval_decision = 'APPROVED'
              AND aa.approval_stage <> NEW.approval_stage
        ) THEN
            RAISE EXCEPTION USING ERRCODE = 'P7415',
                MESSAGE = 'SAME_ACTOR_MULTI_STAGE_FORBIDDEN';
        END IF;
    END IF;

    RETURN NEW;
END;
$$;

CREATE TRIGGER adjustment_approval_guard
BEFORE INSERT ON adjustment_approval
FOR EACH ROW
EXECUTE FUNCTION trg_adjustment_approval_guard();
```

## 7.6 Approval after-insert state advancement trigger

This is the deterministic automation that advances status after approvals come in.

```sql
CREATE OR REPLACE FUNCTION trg_adjustment_approval_after_insert_advance()
RETURNS trigger
LANGUAGE plpgsql
AS $$
DECLARE
    v_adj adjustment_instruction%ROWTYPE;
    v_quorum BOOLEAN;
    v_maker_checker_ok BOOLEAN;
BEGIN
    SELECT * INTO v_adj
    FROM adjustment_instruction
    WHERE adjustment_id = NEW.adjustment_id
    FOR UPDATE;

    IF NOT FOUND THEN
        RETURN NULL;
    END IF;

    -- If any rejection exists and adjustment is still in approval flow, mark REJECTED
    IF fn_adjustment_has_rejection(NEW.adjustment_id)
       AND v_adj.adjustment_status IN ('PENDING_APPROVAL','APPROVED_PENDING_COOLING') THEN
        UPDATE adjustment_instruction
           SET adjustment_status = 'REJECTED',
               updated_at = now()
         WHERE adjustment_id = NEW.adjustment_id;
        RETURN NULL;
    END IF;

    -- Check quorum + maker-checker validity
    v_quorum := fn_adjustment_approval_quorum_satisfied(NEW.adjustment_id);
    v_maker_checker_ok := fn_adjustment_maker_checker_valid(NEW.adjustment_id);

    IF v_quorum AND NOT v_maker_checker_ok THEN
        RAISE EXCEPTION USING ERRCODE = 'P7411',
            MESSAGE = 'MAKER_CHECKER_VIOLATION';
    END IF;

    IF v_quorum AND v_maker_checker_ok THEN
        -- Advance based on cooling
        IF v_adj.cooling_required THEN
            IF v_adj.adjustment_status = 'PENDING_APPROVAL' THEN
                UPDATE adjustment_instruction
                   SET adjustment_status = 'APPROVED_PENDING_COOLING',
                       updated_at = now()
                 WHERE adjustment_id = NEW.adjustment_id;
            END IF;
        ELSE
            IF v_adj.adjustment_status = 'PENDING_APPROVAL' THEN
                UPDATE adjustment_instruction
                   SET adjustment_status = 'ELIGIBLE_TO_EXECUTE',
                       updated_at = now()
                 WHERE adjustment_id = NEW.adjustment_id;
            END IF;
        END IF;
    END IF;

    RETURN NULL;
END;
$$;

CREATE TRIGGER adjustment_approval_after_insert_advance
AFTER INSERT ON adjustment_approval
FOR EACH ROW
```

## 7.7 Cooling-expiry promotion function (scheduled job safe)

Use a controlled function (called by scheduler) to move `APPROVED_PENDING_COOLING` → `ELIGIBLE_TO_EXECUTE`.

```sql
CREATE OR REPLACE FUNCTION fn_promote_adjustments_after_cooling(p_limit INTEGER DEFAULT 1000)
RETURNS INTEGER
LANGUAGE plpgsql
AS $$
DECLARE
    v_count INTEGER := 0;
BEGIN
    WITH candidates AS (
        SELECT adjustment_id
        FROM adjustment_instruction
        WHERE adjustment_status = 'APPROVED_PENDING_COOLING'
          AND cooling_required = TRUE
          AND cooling_until IS NOT NULL
          AND now() >= cooling_until
        ORDER BY cooling_until
        LIMIT p_limit
        FOR UPDATE SKIP LOCKED
    )
    UPDATE adjustment_instruction ai
       SET adjustment_status = 'ELIGIBLE_TO_EXECUTE',
           updated_at = now()
      FROM candidates c
     WHERE ai.adjustment_id = c.adjustment_id;

    GET DIAGNOSTICS v_count = ROW_COUNT;
    RETURN v_count;
END;
$$;
```

## 7.8 Execution insert guard (approvals + cooling + state)

```sql
CREATE OR REPLACE FUNCTION trg_adjustment_execution_insert_guard()
RETURNS trigger
LANGUAGE plpgsql
AS $$
DECLARE
    v_adj adjustment_instruction%ROWTYPE;
    v_quorum BOOLEAN;
    v_exec_exists BOOLEAN;
BEGIN
    SELECT * INTO v_adj
    FROM adjustment_instruction
    WHERE adjustment_id = NEW.adjustment_id
    FOR UPDATE;

        RAISE EXCEPTION USING ERRCODE = 'P7420', MESSAGE = 'ADJUSTMENT_NOT_EXECUTABLE';
    END IF;

    IF v_adj.adjustment_status NOT IN ('ELIGIBLE_TO_EXECUTE','APPROVED_PENDING_COOLING') THEN

            MESSAGE = 'ADJUSTMENT_APPROVALS_INCOMPLETE';

        RAISE EXCEPTION USING ERRCODE = 'P7411',
            MESSAGE = 'MAKER_CHECKER_VIOLATION';
    END IF;

    -- cooling gate
    IF v_adj.cooling_required
       AND v_adj.cooling_until IS NOT NULL
       AND now() < v_adj.cooling_until THEN
        RAISE EXCEPTION USING ERRCODE = 'P7421',
            MESSAGE = 'COOLING_OFF_PERIOD_NOT_EXPIRED';
    END IF;

    -- prevent concurrent active executions (simplified)
    SELECT EXISTS (
        SELECT 1
        FROM adjustment_execution ae
        WHERE ae.adjustment_id = NEW.adjustment_id
          AND ae.execution_status IN ('PENDING','SENT','ACKED')
    ) INTO v_exec_exists;

    IF v_exec_exists THEN
        RAISE EXCEPTION USING ERRCODE = 'P7423',
            MESSAGE = 'EXECUTION_ALREADY_IN_PROGRESS_OR_TERMINAL';
    END IF;

    RETURN NEW;
END;
$$;
```

## 7.10 Execution status update guard + parent-state terminal projection

This prevents illegal execution state transitions and projects final execution result to `adjustment_instruction`.

```sql
CREATE OR REPLACE FUNCTION trg_adjustment_execution_update_guard_and_project()
RETURNS trigger
LANGUAGE plpgsql
AS $$
DECLARE
    v_adj adjustment_instruction%ROWTYPE;
BEGIN
    -- Legal execution state transitions
    IF NEW.execution_status IS DISTINCT FROM OLD.execution_status THEN
        PERFORM 1 FROM (
            VALUES
              ('PENDING','SENT'),
              ('PENDING','FAILED'),
              ('PENDING','CONTAINED'),
              ('SENT','ACKED'),
              ('SENT','FAILED'),
              ('SENT','CONTAINED'),
              ('ACKED','FINAL_CONFIRMED'),
              ('ACKED','FAILED'),
              ('ACKED','CONTAINED')
        ) AS t(from_state, to_state)
        WHERE t.from_state = OLD.execution_status
          AND t.to_state = NEW.execution_status;

        IF NOT FOUND THEN
            RAISE EXCEPTION USING ERRCODE = 'P7423',
                MESSAGE = format('INVALID_EXECUTION_STATE_TRANSITION: %s -> %s', OLD.execution_status, NEW.execution_status);
        END IF;
    END IF;

    NEW.updated_at := now();

    -- Project execution terminal outcome into adjustment instruction status (AFTER part simulated in BEFORE + update)
    IF NEW.execution_status IN ('FINAL_CONFIRMED','FAILED','CONTAINED') THEN
        SELECT * INTO v_adj
        FROM adjustment_instruction
        WHERE adjustment_id = NEW.adjustment_id
        FOR UPDATE;

        IF FOUND THEN
            IF NEW.execution_status = 'FINAL_CONFIRMED' THEN
                UPDATE adjustment_instruction
                   SET adjustment_status = 'EXECUTED',
                       updated_at = now()
                 WHERE adjustment_id = NEW.adjustment_id
                   AND adjustment_status = 'EXECUTION_IN_PROGRESS';
            ELSE
                -- For FAILED/CONTAINED, keep adjustment in EXECUTION_IN_PROGRESS or move back to ELIGIBLE?
                -- Conservative approach: keep in EXECUTION_IN_PROGRESS and require explicit operator decision / new execution attempt
                -- Alternative: move to ELIGIBLE_TO_EXECUTE if policy says retryable.
                NULL;
            END IF;
        END IF;
    END IF;

    RETURN NEW;
END;
$$;

CREATE TRIGGER adjustment_execution_update_guard_and_project
BEFORE UPDATE ON adjustment_execution
FOR EACH ROW
EXECUTE FUNCTION trg_adjustment_execution_update_guard_and_project();
```

> Note: For `FAILED` / `CONTAINED`, I intentionally do **not** auto-project to `REJECTED` because execution failure ≠ governance rejection. Usually you want another execution attempt or a distinct operator decision. Keep that explicit.

# 11) Concurrency / race-condition notes (important)

This is where systems usually break under pressure.

## 11.1 Use `FOR UPDATE` on parent adjustment row

All approval insertion and execution-start logic should lock the parent adjustment row.

## 11.2 Use `SKIP LOCKED` for batch promoters

Cooling/expiry jobs must use `FOR UPDATE SKIP LOCKED` to avoid duplicate workers racing.

## 11.3 Avoid double execution starts

Execution insert guard checks active execution rows before insert and locks the adjustment row.

## 11.4 Idempotent command layer

Service/API should supply idempotency keys for:

* submit for approval
* approve stage
* create execution attempt

DB triggers help, but command idempotency is still needed.

# 5) Final Boss — Rail Re-entry Logic (Duplicate Detection)

## Verdict: this is the real production gate

This is the most important critique in the whole note.

If an adjustment is re-queued naively using the same partner-facing reference, many rails will:

* reject as duplicate
* ignore silently
* reconcile to original attempt incorrectly
* create ambiguous state explosion

You need a **two-layer reference strategy**.

---

# Symphony Adjustment Reference Strategy (Canonical)

## Core design principle

Maintain **one immutable business lineage ID**, but allow **new rail-facing dispatch references** per adjustment execution attempt.

### Layer A — Business lineage (stable, internal/audit)

Never changes across original + adjustments:

* `lineage_id` (or original `instruction_id`)
* `customer_intent_id` (optional)
* `correlation_id` (global lineage correlation)

This is the anchor for evidence reconstruction.

### Layer B — Rail dispatch identity (per external send)

Changes for each dispatch attempt:

* `partner_client_reference`
* `dispatch_attempt_reference`
* `adapter_message_id`

This is what prevents rail duplicate collisions.

---

## What to send to rails (recommended)

For an approved adjustment execution, generate a **new partner-facing reference**.

### Example strategy

If original was:

* `SYM-2026-000123`

Adjustment execution attempt becomes:

* `SYM-2026-000123-A1` (adjustment #1)
* `SYM-2026-000123-A1-E1` (execution attempt 1)
* `SYM-2026-000123-A1-E2` (execution retry attempt 2)

Or a stricter machine form:

* `L:<lineage_short>|ADJ:01|EX:01`

The exact format depends on rail length/charset limits.

## Critical requirement

The adapter must carry **both**:

1. partner-facing unique dispatch reference (for the rail)
2. internal lineage/correlation (for Symphony evidence)

If rail only allows one field, adapter must maintain a deterministic mapping table and preserve the original lineage in evidence.

---

## Why not reuse original reference?

Because partner duplicate detection often keys on:

* client reference
* amount
* recipient
* date window

Reusing the same reference for a corrective/adjustment flow can cause:

* false duplicate rejection
* silent acceptance but mapped to original attempt
* reconciliation ambiguity

**Adjustment re-entry must be a new external message, not a replay masquerading as the original.**

---

## 7.4 Strategy families (allowed)

### A) Fresh deterministic reference (recommended default for adjustments)

Generate a new unique partner ref for `ADJUSTMENT_REENTRY`, with lineage encoded indirectly.

Best when:

* rail strongly dedupes on client ref
* related reference field exists (or internal mapping is enough)

### B) Suffix-based derivation (only if adapter supports)

Example:

* original: `SYM240224ABC123`
* adjustment: `SYM240224ABC123A1`
* retry: `...E2`

Use only when length limits permit and partner accepts suffix variation.

### C) Tokenized compact derivation (length-constrained rails)

Use a compact encoded token from lineage + adjustment + execution numbers.

Example:

* `S9K2M4A1E1`

Must be registry-backed and evidence-linked.

# 13) SQL Enforcement (Triggers / Guards)

## 13.1 Registry immutability trigger (after dispatch)

```sql id="n3wv8t"
CREATE OR REPLACE FUNCTION trg_dispatch_reference_registry_immutability()
RETURNS trigger
LANGUAGE plpgsql
AS $$
DECLARE
    v_locked BOOLEAN;
BEGIN
    v_locked := OLD.status IN ('DISPATCHED','ACKED','FINAL_CONFIRMED','FAILED','CONTAINED','VOIDED');

    IF v_locked THEN
        IF NEW.partner_client_reference IS DISTINCT FROM OLD.partner_client_reference
           OR NEW.partner_related_reference IS DISTINCT FROM OLD.partner_related_reference
           OR NEW.reference_strategy_name IS DISTINCT FROM OLD.reference_strategy_name
           OR NEW.reference_strategy_version IS DISTINCT FROM OLD.reference_strategy_version
           OR NEW.adapter_capability_version IS DISTINCT FROM OLD.adapter_capability_version
           OR NEW.strategy_input_hash IS DISTINCT FROM OLD.strategy_input_hash
           OR NEW.strategy_output_hash IS DISTINCT FROM OLD.strategy_output_hash
        THEN
            RAISE EXCEPTION USING
                ERRCODE = 'P7501',
                MESSAGE = 'DISPATCH_REFERENCE_IMMUTABLE_AFTER_LOCK';
        END IF;
    END IF;

    NEW.updated_at := now();
    RETURN NEW;
END;
$$;

CREATE TRIGGER dispatch_reference_registry_immutability
BEFORE UPDATE ON dispatch_reference_registry
FOR EACH ROW
EXECUTE FUNCTION trg_dispatch_reference_registry_immutability();
```

```md
# TSK-HARD-094 PLAN

## Objective
Implement Offline Safe Mode so Symphony degrades safely during load-shedding / network outages: it preserves deterministic containment and evidence integrity while pausing unsafe outbound dispatch until dependencies recover.

## Scope
### In Scope
- Offline Safe Mode detection and state (system-level and per-adapter/rail).
- Dependency health signals: rail reachability, DNS/TLS failure rates, signing service reachability (if required), policy verification availability.
- Safe-mode behavior:
  - pause outbound dispatch (default)
  - allow ingestion (configurable) into contained states
  - durable local queue for deferred actions
- Recovery behavior:
  - deterministic replay with idempotency enforcement
  - evidence events for replay outcomes

### Out of Scope
- Full offline settlement/payment processing.
- Cryptographic signing tier changes (TSK-HARD-096).
- DR recovery package / PKA work (later waves).

## Implementation Steps
1. Define “offline safe mode” triggers:
   - threshold-based reachability failures
   - dependency unavailability timers
2. Add system/adaptor status model:
   - `ONLINE`, `DEGRADED`, `OFFLINE_SAFE_MODE`, `RESUME_PENDING`
3. Implement dispatch pause:
   - dispatch scheduler checks status and blocks sending in OFFLINE_SAFE_MODE
4. Implement durable queue:
   - persist deferred actions/events locally (DB-backed queue or outbox)
5. Implement recovery replay:
   - replay queued items deterministically
   - enforce idempotency keys
   - emit evidence per replay batch
6. Add operator override workflow:
   - explicit “resume” action with reason
   - emit evidence and lock to audit
7. Write tests:
   - activation on dependency failure
   - no outbound dispatch during safe mode
   - replay idempotency and correctness
8. Emit evidence artifacts:
   - offline start/end
   - queued counts
   - replay results

## Design / Invariant Constraints
- Offline mode must never cause unsafe dispatch or retries.
- Evidence must clearly distinguish “queued during offline” vs “executed after recovery.”
- No silent fallback to unsigned outputs where signing is required.
```
